# BOOTH BUDDY ｜ 社内共有（デプロイ）ガイド

「自分のPCのlocalhostでしか開けない」状態から、**他の人にも使ってもらえる**状態にするための手順です。
非エンジニアでも進められるように書いています。エンジニアに渡す場合はこのファイルごと渡せばOKです。

---

## 0. このアプリの状態

- **モデル**：`claude-sonnet-4-6`（2026年6月時点の現行。古いモデル名だと「返事が来ない＝止まって見える」原因になる）。
- **合言葉（パスワード）**：`.env` または環境変数の `APP_PASSWORD` に文字を入れると、その合言葉を知っている人だけが使える。空ならゲートなし。
- **`/health` ページ**：ブラウザで `（アプリのURL）/health` を開くと、モデル名・知識件数・APIキー有無・合言葉の要否が見える（動作確認用）。
- **`.gitignore`**：`.env`（APIキー）と `node_modules` を誤って公開しないようにしてある。

---

## 1. まず手元で動くか確認する（5分）

1. このフォルダでターミナル（PowerShell）を開く。
2. `npm install`
3. `.env.example` をコピーして `.env` を作り、APIキーを入れる。
4. `npm start` → `http://localhost:3000` が出れば成功。
5. うまくいかない時は **http://localhost:3000/health** を開く。
   - `hasApiKey: false` → `.env` のAPIキー未設定。
   - 返事が来ない → 黒い画面の赤いエラー文を確認（多くはAPIキー失効・残高切れ・モデル名）。

> ここで動けばアプリ本体は完成。あとは「どう他の人に届けるか」だけです。

## 2. 共有のしかた：2つの選び方

| | 方法A：社内LAN（同じ建物内） | 方法B：クラウド（Render） |
|---|---|---|
| 手軽さ | ◎ いちばん簡単。クラウド登録不要 | ○ 登録と数ステップ必要 |
| どこから使える | 同じ事務所のWi-Fi内だけ | どこからでも（スマホ可） |
| URL | `http://192.168.x.x:3000` | `https://○○○.onrender.com` |
| マイク音声入力 | ⚠️ 使えないことが多い（後述） | ◎ 使える |
| 費用 | 0円（API従量課金のみ） | 無料枠あり（API従量課金は別） |

**マイクの注意**：ブラウザの音声入力（🎤）は、安全な接続（https または localhost）でしか動きません。
方法A（`http://192.168…`）では文字チャットと読み上げは動きますが、**マイク入力は無効**です。マイクも使うなら方法B（Render＝https）を。

---

## 3. 方法A：社内LANで共有（同じ事務所内）

1. `.env` に合言葉を設定（共有時は必須）：`APP_PASSWORD=oyakata2026`
2. 「常にONにしておく1台」で `npm start` を実行したままにする。
3. そのPCのIPを調べる（PowerShellで `ipconfig` → `IPv4 アドレス`、例 `192.168.1.23`）。
4. 他の人は同じWi-Fiから **`http://192.168.1.23:3000`** を開く。最初に合言葉を聞かれる。
5. つながらない時：そのPCのファイアウォールで Node.js / ポート3000 の受信を許可。

> PCを再起動・スリープすると止まります。常時使うなら方法Bが楽です。

---

## 4. 方法B：Render で公開（推奨・どこからでも・マイク可）

Render は、このNodeアプリをほぼそのまま動かせる無料枠ありのサービスです。
流れは「GitHubにコードを置く → Renderがそれを読んで動かす」。

### ステップ1：GitHubにコードを置く
1. github.com でアカウント作成（無料）。
2. 「＋ → New repository」。名前 `booth-buddy-app`、**Private（非公開）**。
3. リポジトリに、このフォルダの中身をアップロード。
   - ⚠️ `.env` は**絶対に上げない**（APIキーが漏れる）。`node_modules` も不要。
   - このフォルダには元々 `.env` と `node_modules` を入れていないので、**まるごと上げてOK**。
   - 手動アップロードより **GitHub Desktop** アプリのほうが `.gitignore` が効いて事故が起きにくく安全。

### ステップ2：Renderと連携
1. render.com でアカウント作成（GitHubでログインすると楽）。
2. 「New → Web Service」→ さっきのリポジトリを接続。
3. 設定：
   - **Build Command**：`npm install`
   - **Start Command**：`npm start`
   - **Instance Type**：Free（まず無料枠）

### ステップ3：環境変数（.envの代わり）
Renderの「Environment」に3つ追加：
- `ANTHROPIC_API_KEY` = あなたのAPIキー
- `ANTHROPIC_MODEL` = `claude-sonnet-4-6`
- `APP_PASSWORD` = 共有する合言葉（例 `oyakata2026`）

### ステップ4：確認
発行された `https://○○○.onrender.com` を開く → 合言葉 → 返事が来ればOK。`/health` でも確認。
無料枠は15分アクセスがないとスリープし、次回起動に30〜60秒かかります（常時即応にしたいなら有料の最小プラン 月7ドル）。

## 5. 共有する前のチェックリスト

- [ ] `APP_PASSWORD` を設定した（無防備な公開はAPI費用の無駄遣い・悪用につながる）。
- [ ] `.env` と `node_modules` を**公開リポジトリに上げていない**。
- [ ] `/health` で `hasApiKey: true` と最新モデルを確認した。
- [ ] Anthropicコンソールで**利用上限（スペンドリミット）**を設定した。
- [ ] 安全・法規・保証に関わる助言は「断定しない／現場・メーカー指示優先」（実運用前に専門家監修推奨）。

## 6. 知識（RAG）を増やす

`knowledge/` フォルダに `.md` を置くだけ（`##` 見出しごとに1チャンク）。
編集後はサーバー再起動、または `（URL）/reload` にPOSTで反映。
社内マニュアル・調色データ・ベテランのコツを足すほど、回答精度が上がります。
